## 硬體、網路要監控的內容
1. **CPU:** 使用率、Load Average (排隊狀況)
2. **RAM:** 剩餘空間、是否用到 Swap、有無 OOM
3. **Disk:** 剩餘空間、I/O 讀寫速度
4. **Network:** 頻寬吞吐量、TCP 連線狀態

這兩個指令是 SRE 常敲的指令，能看 CPU 與 Memory 的狀態：
- 敲下 `top` (看 CPU 與系統負載)
- 敲下 `free -m` (看記憶體)
## 作業系統的垂直層次 (單機視角)

在 Linux 的世界裡，權限與資源是被嚴格分層的。面試官考這個，是想知道你懂不懂「效能瓶頸是怎麼產生的」。

1. **硬體層 (Hardware Layer)：**
    - **內容：** CPU、記憶體、硬碟 (Disk)、網卡 (NIC)。
    - **角色：** 真正的苦力，所有的計算與資料儲存都在這裡發生。

2. **核心空間 (Kernel Space)：**
    - **內容：** Linux Kernel 本身、硬體驅動程式、記憶體管理系統、**網路協議棧 (TCP/IP Stack)**。
    - **角色：** 大管家，擁有最高權限。只有 Kernel 能直接控制硬體。它決定哪個 Process 可以用多少 CPU，並負責把網路封包拆解。

3. **用戶空間 (User Space)：**
    - **內容：** 所有的應用程式。包括你寫的 Python 腳本、Nginx、資料庫、甚至是常用的指令如 `ls`, `top`, `bash`。
    - **角色：** 普通員工。它們**沒有權限**直接讀寫硬碟或發送網路封包。

4. **邊界：系統呼叫 (System Call / Syscall)：**
    - **核心觀念：** 這是 User Space 與 Kernel Space 之間的「海關」。當你的 Nginx 想要讀取硬碟裡的一張圖片，它必須發起 `read()` 這個 System Call，拜託 Kernel 去硬碟拿，Kernel 拿完後再把資料複製回 User Space 給 Nginx。
    - **⭐ 面試考點 (效能瓶頸)：** 穿越這道海關是非常耗費 CPU 資源的（稱為 Context Switch 上下文切換）。如果一個程式設計不良，瘋狂地發起 System Call，就算 CPU 使用率飆高，系統其實沒做多少正事，全在處理切換。
## 網路的水平層次 (通訊視角)

教科書會教 OSI 7 層模型，但 SRE 實戰與面試中，我們更常討論 **TCP/IP 4 層模型**，因為它才符合 Linux Kernel 真正的實作方式。

1. **應用層 (Application Layer) -> 對應 OSI 的 L7：**
    - **協定：** HTTP, HTTPS, DNS, SSH, FTP。
    - **特徵：** 這是「資料」本身的格式。比如 HTTP 規定了 Header 和 Body 怎麼寫。
    - **位置：** 發生在 **User Space**（你的瀏覽器或 Nginx 負責處理）。

2. **傳輸層 (Transport Layer) -> 對應 OSI 的 L4：**
    - **協定：** TCP (可靠、要交握), UDP (不可靠、射後不理)。
    - **特徵：** 負責「端到端」的通訊，也就是透過 **Port (通訊埠)** 把資料精準送到指定的 Process 上。
    - **位置：** 發生在 **Kernel Space**。
        
3. **網路層 (Internet Layer) -> 對應 OSI 的 L3：**
    - **協定：** IP (IPv4, IPv6), ICMP (Ping 用的協定)。
    - **特徵：** 負責「路由 (Routing)」，透過 **IP 位址** 在茫茫網海中找到目標伺服器。
    - **位置：** 發生在 **Kernel Space**。
        
4. **網路接口層 (Link/Network Access Layer) -> 對應 OSI 的 L2/L1：**
    - **協定：** Ethernet (乙太網), MAC 位址, ARP。
    - **特徵：** 負責把數位資料轉成電信號或光訊號，傳遞給實體網路線路上的下一個節點 (Router 或 Switch)。
    - **位置：** 網卡驅動程式 (Kernel) + 實體網卡 (Hardware)。
## 結合兩者 (高頻面試題火力展示)

面試官最愛問的經典題：**「請問從一台伺服器的 Nginx 發送一個 HTTP 回應給客戶端，資料是怎麼從記憶體流向網路線的？」**

如果你能用「層次」來回答，代表你完全懂了：
1. **(User Space - L7)：** Nginx (Process) 準備好了 HTTP Response 資料，存放在自己的 User Space 記憶體中。
2. **(Syscall)：** Nginx 發起 `write()` 或 `send()` System Call，透過 Socket 介面把資料交給 Kernel。
3. **(Kernel Space - L4)：** Kernel 的 TCP 模組接手，把 HTTP 資料切塊，加上 TCP Header (包含 Source Port 80, Destination Port 隨機)。
4. **(Kernel Space - L3)：** IP 模組接手，加上 IP Header (包含 Source IP, Destination IP)。此時這包資料被稱為「封包 (Packet)」。
5. **(Kernel Space - L2)：** ARP 模組查表，加上 MAC Header (下一跳路由器的 MAC 位址)。
6. **(Hardware - L1)：** Kernel 觸發網卡 (NIC) 的硬體中斷，網卡將記憶體中的封包轉為電信號，透過實體網路線發送出去。
