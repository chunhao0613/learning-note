# 核心網路協定與運作機制

了解工具後，你需要知道這些工具抓出來的「協定」在幹嘛。這是你未來處理 K8s 網路問題的基石。
## SRE 網路協定與排障速查表

| **層級** | **協定**     | **運作核心**    | **SRE 實戰關鍵 (為什麼要懂)**                       | **常用 Linux 工具**                           |
| ------ | ---------- | ----------- | ------------------------------------------ | ----------------------------------------- |
| **L2** | **ARP**    | IP 換 MAC 位址 | **VIP 飄移失敗：** Keepalived 切換後快取未更新，導致流量走錯路。 | `ip neigh`, `tcpdump arp`                 |
| **L3** | **IP**     | 決定封包路徑      | **路由導向：** 雲端 VPC 或 K8s 子網路由設錯，封包會直接迷路。     | `ip route`, `mtr`, `ping`                 |
| **L3** | **ICMP**   | 狀態回報與報錯     | **基礎連通：** 診斷節點是否存活，但需注意雲端防火牆常禁 Ping。       | `ping`, `mtr`                             |
| **L4** | **TCP**    | 三方交握/可靠傳輸   | **連線管理：** 處理 `TIME_WAIT` 過多、連線數爆滿及核心參數調優。  | `ss -tlnp`, `netstat`, `tcpdump`          |
| **L4** | **UDP**    | 快速但不保證到達    | **基建依賴：** DNS 查詢、Log 傳遞或監控數據，丟包會導致監控斷線。    | `nc -uvz`, `ss -ulnp`                     |
| **L7** | **DNS**    | 名稱轉 IP      | **服務解法：** K8s 內部通訊核心。若 CoreDNS 故障，整個叢集停擺。  | `dig`, `nslookup`, `cat /etc/resolv.conf` |
| **L7** | **HTTP/S** | 請求/回應格式     | **效能指標：** 監控 5xx 錯誤碼判斷後端狀態，管理 SSL 憑證效期。    | `curl -Iv`, `Nginx logs`                  |
| **L7** | **TLS**    | 加密與驗證       | **安全事故：** 憑證過期會造成 P0 級災難；握手過久會增加延遲。        | `openssl s_client -connect`               |

---

### SRE 除錯三部曲 (Bottom-Up)

1. **檢查 L3/L4 (通不通)：** 用 `ping` 或 `nc` 確認網路路徑與 Port 是否有開。
2. **檢查 L7 (對不對)：** 用 `curl` 確認應用程式回傳的狀態碼與內容。
3. **封包分析 (深層原因)：** 若邏輯不明，直接上 `tcpdump` 抓包看握手過程。
## 雲原生與容器網路排障地圖

| **核心技術 / 機制**                                          | **運作原理與白話文比喻**                                                                                                                                                                                                             | **SRE 實戰排障場景 (為什麼會壞)**                                                                                                                          | **必備進階除錯指令**                                                                                                         |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **Network Namespaces**<br><br>  <br><br>(網路命名空間)       | **空間摺疊與隔離：**<br><br>  <br><br>讓 Linux 核心在同一台機器上，切分出多個完全獨立的網路環境（獨立的網卡、IP、路由表、Port）。<br><br>  <br><br>👉 _比喻：一棟辦公大樓（實體機）被切成多間獨立辦公室（容器），彼此看不到對方的內部網路。_                                                                        | **場景：** 容器內的服務明明啟動了，但宿主機 (Host) 卻連不到。<br><br>  <br><br>**排障邏輯：** 必須「穿牆」進入該容器的專屬空間，才能看到它真實的網路狀態，用外面宿主機的 `ss -tlnp` 是抓不到容器內的 80 Port 的。           | `ip netns list` (列出空間)<br><br>  <br><br>`nsenter -t <PID> -n netstat -antp` (⭐ SRE 神技：穿牆進入容器空間執行檢測)                  |
| **Veth Pair & Bridge**<br><br>  <br><br>(虛擬網線與交換機)     | **打通空間的橋樑：**<br><br>  <br><br>`veth pair` 是一條虛擬網路線，一頭插在容器內，一頭插在宿主機上。而 `Bridge` (如 `docker0` 或 `cni0`) 則是把所有虛擬線路接在一起的交換機。<br><br>  <br><br>👉 _比喻：把各個獨立辦公室的電腦，牽線拉到走廊上的同一台 Switch 上。_                                        | **場景：** 同一台 Node 上的 Pod A 無法連線到 Pod B。<br><br>  <br><br>**排障邏輯：** 檢查虛擬網卡有沒有配對成功，或者 Bridge 的 MTU (最大傳輸單元) 設定是否一致，導致大封包被中途丟棄。                     | `ip link show` (看虛擬網卡)<br><br>  <br><br>`brctl show` (看 Bridge 上插了哪些網卡)<br><br>  <br><br>`ip -d link show type veth` |
| **NAT (DNAT / SNAT)**<br><br>  <br><br>(網路位址轉換)        | **封包變臉魔術：**<br><br>  <br><br>**SNAT (來源轉換)：** 容器連外網時，把虛擬 IP 換成實體機 IP。<br><br>  <br><br>**DNAT (目的轉換)：** 外網連進來時，把實體機 Port 轉發給容器內部 IP。<br><br>  <br><br>👉 _比喻：公司總機（NAT）幫內部員工（容器）打外線，或把外部客戶來電轉接給特定分機。_                       | **場景：** 外部使用者抱怨「連線被拒絕 (Connection Refused)」，但你在容器內測試卻正常。<br><br>  <br><br>**排障邏輯：** 檢查 iptables 的 NAT 表，看 Port Forwarding (通訊埠轉發) 的規則是不是寫錯或遺失了。 | `iptables -t nat -L -n -v`<br><br>  <br><br>`conntrack -L` (查看目前 NAT 正在追蹤的連線狀態)                                      |
| **CNI & Overlay Network**<br><br>  <br><br>(跨節點覆蓋網路)   | **跨機通訊的任意門：**<br><br>  <br><br>當 Node 1 的 Pod 要跟 Node 2 的 Pod 講話時，實體網路根本不認識 Pod 的虛擬 IP。CNI (如 Flannel/Calico) 會把虛擬封包「塞進」實體封包裡 (VXLAN 封裝)，或是動態修改路由表 (BGP)。<br><br>  <br><br>👉 _比喻：把一封公司內部信件，裝進中華郵政的標準信封裡寄到另一棟大樓，對方收到後再拆開。_ | **場景：** 同一台機器的 Pod 互通正常，但跨機器的 Pod 互叫 API 卻全部 Timeout。<br><br>  <br><br>**排障邏輯：** 檢查 Node 之間的 UDP Port (VXLAN 常用) 是不是被外部硬體防火牆擋住了，導致「套娃」封包送不過去。   | `ip -d link show flannel.1` (檢查隧道網卡)<br><br>  <br><br>`route -n` (檢查跨節點路由)                                           |
| **Kube-proxy & Service**<br><br>  <br><br>(K8s 內部負載均衡) | **動態轉發樞紐：**<br><br>  <br><br>K8s 中的 Service VIP (ClusterIP) 其實是個假 IP，背後沒有實體網卡。`kube-proxy` 會在每台機器上下達 iptables 或 IPVS 規則，當封包打向 VIP 時，隨機轉發給背後存活的 Pod。<br><br>  <br><br>👉 _比喻：一個永遠不變的客服專線號碼 (VIP)，背後自動幫你轉接給目前有空的客服人員 (Pod)。_ | **場景：** 打 API 時好時壞 (有時候 200，有時候 502)。<br><br>  <br><br>**排障邏輯：** 背後可能有 3 個 Pod，其中 1 個死了但沒有從 Service 的名單中剔除。必須檢查 Endpoint 列表與 IPVS 轉發規則。         | `kubectl get endpoints`<br><br>  <br><br>`ipvsadm -ln` (⭐ 大規模 K8s 必備：查看 IPVS 負載均衡狀態)                                 |

1. **L7 / L4 外部入口：** 先用 `curl` 或 `dig` 檢查 DNS 與 Ingress/Nginx 狀態。
2. **L3 / L4 節點轉發 (30% 範圍)：** 檢查 `iptables` NAT 表與 `kube-proxy` (DNAT 是否正常運作)。
3. **L2 / Namespace 內部 (30% 範圍)：** 用 `nsenter` 穿牆進去容器看 `ss -tlnp`，確認應用程式真的有在監聽。
